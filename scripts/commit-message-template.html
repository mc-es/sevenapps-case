A well-structured commit message enhances code readability and helps maintain a clear project history. Follow these rules for consistency and clarity.

Structure

Each message should follow:

<type>(<scope>): <subject>
type
Purpose of the change, chosen from a fixed set (see below).
scope
Area affected. Use a short noun in kebab-case (e.g., auth-service).
subject
Concise imperative sentence describing the change.

1. Type

Use lowercase. Pick one:

  • feat: new feature
  • fix: bug fix
  • change: non-breaking changes
  • chore: maintenance (refactor, deps, build)
  • docs: documentation only
  • style: formatting, missing semicolons, etc.
  • perf: performance improvement
  • test: add or update tests

2. Scope

Short, descriptive noun in kebab-case. Examples:

fix(auth-service): correct token validation
feat(ui-button): add loading spinner

3. Subject

✓
Write in imperative mood (command-like): add, fix, update.
✓
Use lowercase; only capitalize proper nouns.
✕
Don’t end with a period .

4. Length

Subject ≤ 100 chars. If needed, add a body after a blank line.

5. Contains a verb

The message must include a verb, e.g.:

feat(ui): add loading spinner
docs(readme): update installation steps

6. Verb tense

✕
Wrong: docs(readme): updated installation steps
✓
Correct: docs(readme): update installation steps

Example commit messages

  • ✓ feat(theme): add theme provider for dynamic theme switching
  • ✓ feat(tailwind): implement Tailwindcss for styling
  • ✓ fix(auth-service): correct token validation
  • ✓ chore(set-up): add ESLint rules for code linting
  • ✓ docs(readme): clarify installation steps
  • ✓ test(unit): add tests for login component
  • ✕ feature(theme): add theme provider for dynamic theme switching
  • ✕ feat: configure i18n-js for TR and EN language support
  • ✕ feat(): add splash screen and app icon
  • ✕ chore(Set-up): add ESLint rules for code linting
  • ✕ docs(file): Add README file with project overview
  • ✕ test(unit): added test cases for login component
